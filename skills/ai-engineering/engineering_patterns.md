# AI Engineering Patterns

Этот документ описывает передовые паттерны работы с автономными агентами для включения их в экосистему (Borsch-core, Mac AI Ecosystem и др.). 
Основано на архитектуре `ai-website-cloner-template` и паттернах из фреймворка `Mastra`.

## 1. Single Source of Truth для правил Агентов (Sync Script)
Содержать системные промпты и правила проекта синхронно для Cursor, Windsurf, Claude Code, Cline и Gemini CLI — невозможно вручную.

**Решение:**
Использовать мастер-файл `AGENTS.md` в корне. Скрипт резолвит импорты вида `@docs/ARCHITECTURE.md` прямо внутрь и генерирует нативные конфиги для каждой IDE автоматически на pre-commit или вручную.

**Bash-скрипт (`scripts/sync-agent-rules.sh`):**
```bash
#!/usr/bin/env bash
set -euo pipefail
REPO_ROOT="$(cd "$(dirname "$0")/.." && pwd)"
SOURCE="$REPO_ROOT/AGENTS.md"

resolve_imports() {
  while IFS= read -r line || [[ -n "$line" ]]; do
    if [[ "$line" =~ ^@(.+)$ ]]; then
      cat "$REPO_ROOT/${BASH_REMATCH[1]}"
    else
      echo "$line"
    fi
  done < "$SOURCE"
}

RESOLVED_CONTENT="$(resolve_imports)"
HEADER="<!-- AUTO-GENERATED from AGENTS.md — do not edit directly. -->"

echo -e "$HEADER\n$RESOLVED_CONTENT" > "$REPO_ROOT/.clinerules"
echo -e "$HEADER\n$RESOLVED_CONTENT" > "$REPO_ROOT/.windsurfrules"
# Для Cursor можно обновить .cursor/rules/project.mdc (согласно формату cursor)
```

## 2. Сниппет "JavaScript Пылесос" (Точный CSS для реверс-инжиниринга)
Для идеального попиксельного UI-прототипирования ИИ не должен угадывать паддинги. ИИ-агент инжектит JavaScript через Browser MCP / Playwright для снятия слепка `getComputedStyle`.

**Сниппет для извлечения Computed Styles и слоев:**
```javascript
(function(selector) {
  const el = document.querySelector(selector);
  if (!el) return null;
  // Основные свойства, влияющие на Layout и UI
  const props = ['color', 'padding', 'margin', 'fontSize', 'fontWeight', 'lineHeight', 'borderRadius', 'display', 'gap', 'backgroundColor', 'boxShadow', 'position', 'zIndex', 'opacity'];
  
  function walk(node, depth) {
    if(depth > 4) return null; // Ограничение глубины
    const cs = getComputedStyle(node);
    const styles = {};
    props.forEach(p => { 
      // Игнорируем дефолтные прозрачные значения
      if (cs[p] && cs[p] !== '0px' && cs[p] !== 'none' && cs[p] !== 'rgba(0, 0, 0, 0)') {
        styles[p] = cs[p];
      } 
    });
    
    return { 
      tag: node.tagName.toLowerCase(), 
      classes: node.className,
      text: node.childNodes.length === 1 && node.nodeType === 3 ? node.textContent.trim() : null,
      styles, 
      children: [...node.children].map(c => walk(c, depth + 1)).filter(Boolean)
    };
  }
  return JSON.stringify(walk(el, 0), null, 2);
})('.target-component');
```

## 3. Оркестрация через Git Worktrees (Анти-Галлюцинации)
Вместо того чтобы агент писал 2000 строк кода и терял контекст, Оркестратор:
1. Выгружает требования в отдельный файл спецификации: `docs/components/Feature.spec.md`.
2. Создает чистый **Git Worktree** (где нет риска сломать основной код во время раздумий).
3. Запускает Sub-Agent'а внутри этого ворктри, передав ему только `.spec.md`.

```bash
# 1. Оркестратор подготавливает изолированную среду
git branch feature-hero
git worktree add ../frontend-feature-hero feature-hero

# 2. Worker Агент пишет код в ../frontend-feature-hero (без влияния на основной репо)

# 3. После завершения билда и QA воркером, Оркестратор в основном репо:
git merge feature-hero
git worktree remove ../frontend-feature-hero
git branch -d feature-hero
```

## 4. Защита от Supply Chain Атак (Min Release Age)
Агенты могут автоматически устанавливать новые пакеты. Чтобы защититься от публикации вредоносных пакетов (как в случае с axios), внедряется карантин на свежие релизы.
- В глобальном `~/.npmrc`: `min-release-age=7`
- Для `uv` (Python) в `uv.toml`: `exclude-newer = "7 days"`

## 5. Протокол Handoffs (Передача контекста)
Агенты ограничены размером контекста. Для сохранения преемственности при завершении сессии или длительной задачи формируется **Handoff-отчет**:
1. **Что сделано**: (Краткое резюме)
2. **Что НЕ сработало**: (Важнейшая часть — тупики, краши, ошибки зависимостей)
3. **Следующий шаг**: (Одно четкое действие)

## 6. Structured Reasoning (Структурированное рассуждение)
Для сложного дебага вместо "свободного потока мыслей" применяется жесткий шаблон:
- **Факты**: Что *абсолютно точно* известно (логи, строки кода).
- **Трассировка**: Пошаговый разбор логики.
- **Гипотезы**: Список проверенных и отброшенных идей.
*Позволяет избежать ситуации, когда агент ходит по кругу.*

## 7. Hooks > Rules (Гарантии вместо пожеланий)
Правило в промпте (например, "будь осторожен с конфигами") агент забудет через 20 минут активного кодинга.
**Hook** — скрипт (git hook, bash wrapper), который работает механически каждый раз. Детерминированные задачи (тесты, проверка форматирования, напоминание записать Handoff) не должны зависеть от "памяти" модели.
*Принцип из Proof Loop: агент не должен сам подтверждать, что его код работает. Это делает скрипт-верификатор.*

## 8. Dynamic Skills Library (Динамическая подгрузка знаний)
**Проблема:** Большие монолитные инструкции (например, огромные воркфлоу или `rules` файлы) перегружают контекст и "размывают" внимание агента.
**Решение:** (Из Mastra). Во время работы агент должен иметь возможность *динамически*, через семантический поиск, подгружать нужные инструкции (Skills) только когда они нужны. 
- **Реализация:** Вместо одного большого `global_workflows/*.md` использовать библиотеку микро-гайдлайнов (`SKILL.md` или `skill/*.md`), чтобы агент через RAG/Knowledge Item читал как реализовать конкретный UI паттерн или написать тест.

## 9. Жёсткое закрепление планов (Plan Persistence)
**Проблема:** Агенты сразу бросаются писать код, часто игнорируя этап вдумчивой декомпозиции. Если их контекст прерывается, они теряют архитектурный план разработки.
**Решение:** (Из Mastra `submit_plan`).
Вводим обязательный этап **до написания кода**: ИИ или Оркестратор пишет архитектурный план и физически сохраняет его на диск (например, `.dev/plans/2026-04-12-auth-feature.md`). 
Только после одобрения этого `.md` плана (человеком или QA агентом), запускается фаза "Build". ИИ обязан сверяться с этим файлом в процессе реализации.

## 10. Предотвращение сна (Caffeinate Workflow)
**Проблема:** При запуске тяжелых длительных Bash-скриптов агентом (симуляции тестов E2E Playwright, сборки Tauri или Docker), macOS может уйти в режим сна, обрывая соединения или останавливая таймеры.
**Решение:** Оборачивать долгие системные скрипты с помощью утилиты `caffeinate`.
```bash
# ИИ вызывает длительный скрипт так:
caffeinate -i npm run build &
```
Это позволит безопасно оставлять агента работать в фоне на продолжительное время.

## 11. Песочница агента (Agent Sandbox)
Команды `/turbo-all` или экспериментальные воркфлоу должны быть ограничены директориями в целях безопасности (принцип наименьших привилегий из Mastra `allowedPaths`). В случаях агрессивного рефакторинга агент должен:
1. Запросить создание отдельной изолированной директории в корне (`.sandbox/` или `worktree`, см. Паттерн 3).
2. Выполнять все bash-манипуляции со включенным флагом YOLO-mode (Auto-approve) **только** в рамках этой директории.

## 12. Agent-Native CLI (Обертка над Bash-хаосом)
**Проблема:** Предоставление ИИ полного доступа к сырым bash-командам (например, сложные пайплайны с флагами или docker-команды) тратит прорву токенов и ведет к галлюцинациям.
**Решение:** (Основано на подходе Android CLI для агентов). Вводим собственный легковесный CLI (например, `eco` или `borsh` wrapper). 
Агент использует детерминированные команды:
- `eco module create [name]`
- `eco test [domain]`
- `eco docs mlx` (динамический подтяг документации в обход Knowledge Cutoff)
Это радикально снижает шанс ошибки и экономит контекст. Команды-обертки пишут разработчики, агенты их только вызывают. Метаданные внутри воркфлоу файла подсказывают агенту, какую `eco`-команду нужно запустить.

## 13. Локальный Agent CI (Быстрый цикл обратной связи)
**Проблема:** Агенты запускают тесты, ломают код и пытаются пересобрать его, тратя время на холодные старты (установку зависимостей) и теряя фокус. В удаленном CI дебаг еще дороже.
**Решение:** Использовать **Agent-CI** (`@redwoodjs/agent-ci`), который запускает GitHub Actions локально, специально для ИИ-агентов.
Отличия от обычного запуска:
1. **~0 ms кэш**: `node_modules` и браузеры маппятся с хоста в контейнер. 
2. **Пауза при падении (`--pause-on-failure`)**: Если step падает, Agent-CI ставит контейнер на паузу. Агент может поправить код в своей IDE, изменения моментально прокинутся в контейнер, и агент просто нажимает "retry" для упавшего шага (а не начинает билд с нуля).
Команда для агента: `npx @redwoodjs/agent-ci run --all --pause-on-failure`
Интеграция с ИИ: Сохранять коммит перед запуском Agent-CI, как точку сохранения "до поломки".
