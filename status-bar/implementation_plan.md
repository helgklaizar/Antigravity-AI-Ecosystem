# План аудита и оптимизации `AntigravityBar`

После проведения аудита исходного кода выявлено несколько критических проблем с производительностью (блокировка Main Thread) и архитектурным распределением (God Object). 

Ниже представлен план рефакторинга и оптимизации, направленный на минимизацию использования ресурсов и улучшение отзывчивости.

## 1. Оптимизация производительности (Критично)

### 1.1 Асинхронный подсчет размера кэша (Блокировка Main Thread)
**Проблема:** В `AppDelegate.updateBarTitle()` синхронно вызывается `api.cacheSize()`, который обходит все файлы в папках `brain`, `conversations` и `html_artifacts` каждую секунду (при поллинге). Если в кэше много файлов, это вызывает микрофризы UI и излишнюю нагрузку на диск/CPU.
**Решение:**
- Запускать обход файловой системы на `background` очереди и обновлять значение реже (например, раз в 60 секунд).
- Кэшировать последнее вычисленное значение в `AntigravityAPI` и моментально отдавать его для отрисовки UI.

### 1.2 Оптимизация поиска демона (Высокая нагрузка на CPU)
**Проблема:** `findActiveDaemon()` вызывается каждые 2 секунды. Внутри он делает `sysctl` для чтения `KERN_PROCARGS2` для 2048 процессов. Это тяжелая операция для периодического поллинга.
**Решение:**
- Ввести кэширование найденного `DaemonInfo`.
- Искать процессы через `sysctl` *только* в том случае, если текущий закэшированный демон перестал отвечать.

### 1.3 Улучшение цикла поллинга (Перекрытие запросов)
**Проблема:** `Timer.scheduledTimer` запускает `fetchAndUpdate()`, который делает асинхронный сетевой запрос, но таймер планирует следующий вызов не дожидаясь завершения предыдущего.
**Решение:**
- Вызывать `scheduleNextPoll()` строго *после* завершения замыкания ответа от демона, предотвращая накопление параллельных запросов при плохой сети.

## 2. Архитектурный рефакторинг (Очистка)

### 2.1 Разгрузка `AppDelegate` (God Object)
**Проблема:** `AppDelegate` содержит бизнес-логику фильтрации меню, хранения истории графиков и форматирования.
**Решение:**
- Вынести массивы истории (`historyCPU`, `historyGPU`, `historyRAM`) в сам `SystemStats`.
- Вынести сложную логику сортировки квот и склейки моделей из `showContextMenu` в отдельный метод.

## 3. Минимизация кода
- Очистка от излишнего дублирования при добавлении в историю графиков (управление массивом).
- Изоляция логики построения меню в отдельные функции для чистоты.
