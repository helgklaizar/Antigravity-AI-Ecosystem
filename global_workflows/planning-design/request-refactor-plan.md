Создание детального плана рефакторинга с крошечными атомарными коммитами через интервью пользователя, с последующим оформлением в виде GitHub Issue. Используй, когда пользователь хочет спланировать рефакторинг, создать RFC на рефакторинг или разбить крупную переработку кода на безопасные, инкрементальные шаги.

Этот навык вызывается, когда пользователь хочет создать запрос на рефакторинг. Пройди по шагам ниже. Ты можешь пропускать шаги, если считаешь их ненужными в данном контексте.

1. **Сбор проблемы**: Попроси пользователя дать длинное, детальное описание проблемы, которую он хочет решить, и любые потенциальные идеи решений.
2. **Исследование кода**: Исследуй репозиторий, чтобы подтвердить его утверждения и понять текущее состояние кодовой базы.
3. **Альтернативы**: Спроси пользователя, рассматривал ли он другие варианты, и сам предложи ему альтернативные подходы.
4. **Интервью по реализации**: Проведи интервью с пользователем о будущей реализации. Будь предельно детальным и тщательным.
5. **Границы скоупа**: Утверди точные границы реализации. Реши, что именно планируется изменить, а что — не трогать.
6. **Покрытие тестами**: Проверь кодовую базу на наличие тестов в изменяемой зоне. Если покрытие недостаточное, спроси пользователя о планах по тестированию.
7. **Атомарные коммиты**: Разбей реализацию на план крошечных коммитов. Помни совет Мартина Фаулера: "Делай каждый шаг рефакторинга настолько маленьким, насколько это возможно, чтобы всегда видеть, что программа работает".
8. **Создание Issue**: Создай GitHub Issue с планом рефакторинга. Используй следующий шаблон:

## Постановка проблемы (Problem Statement)
Проблема, с которой столкнулся разработчик, с его точки зрения.

## Решение (Solution)
Решение проблемы, с точки зрения разработчика.

## Коммиты (Commits)
ДЛИННЫЙ, подробный план реализации. Напиши план простым языком, разбивая реализацию на мельчайшие возможные коммиты. Каждый коммит должен оставлять кодовую базу в рабочем состоянии.

## Документ решений (Decision Document)
Список принятых архитектурных и реализационных решений. Сюда можно включить:
- Модули, которые будут созданы/изменены
- Интерфейсы модулей, которые подвергнутся модификации
- Технические уточнения от разработчика
- Архитектурные решения
- Изменения схемы (БД и т.д.)
- Контракты API
- Специфические взаимодействия

НЕ включай сюда специфические пути к файлам или сниппеты кода. Они могут очень быстро устареть.

## Решения по тестированию (Testing Decisions)
Список принятых решений по тестированию. Включает:
- Описание того, что делает тест хорошим (тестирование внешнего поведения, а не деталей реализации)
- Какие модули будут протестированы
- Аналоги в проекте (похожие типы тестов, уже существующие в кодовой базе)

## Вне рамок (Out of Scope)
Описание того, что выходит за рамки данного рефакторинга (что мы не трогаем).

## Дополнительные заметки (Further Notes - optional)
Любые дополнительные заметки по рефакторингу.
