# AI UI Prototyping (Spec-Driven Orchestrator)

**Глобальный Паттерн:** Вместо прямой кодогенерации ("сделай UI") используется многоэтапный подход с написанием формального контракта `*.spec.md` и последующим делегированием задач в параллельные git-ветки (Worktrees). Это полностью исключает галлюцинирование ИИ и потерю контекста на сложных интерфейсах.

## Шаг 1: Разведка (Reconnaissance) и Извлечение
1. Если есть референс — используй JS-сниппеты или браузер для извлечения абсолютных значений `getComputedStyle`, цветов (oklch/hex), шрифтов, свойств hover, анимаций скролла (IntersectionObserver, scroll-snap) и состояний слоя.
2. Не угадывай классы ("наверное тут padding-4"), извлекай точные пиксельные/rem значения для Foundation (Базы).
3. Установи глобальные CSS-токены, шрифты и скачай/сгенерируй нужные ассеты (иконки, SVG) до старта работы над компонентами.

## Шаг 2: Генерация Спецификации (*.spec.md)
1. Перед написанием любого кода, вы (Оркестратор) обязаны создать подробный Маркдаун-контракт (например: `docs/research/components/HeroSection.spec.md`).
2. В файле строго опишите:
   - DOM иерархию.
   - Точные Computed Styles (отступы, тени, z-index, позиционирование).
   - Интерактивную модель: что вызывает смену состояния (Click-driven, Scroll-driven, Time-driven).
   - Изменения (State A vs State B) и длительность транзишенов.
   - Базовый контент и адаптивность (Desktop 1440, Tablet 768, Mobile 390).

## Шаг 3: Параллельная Застройка (Worker Agents)
1. Разбейте сложный UI на под-компоненты. Правило: спецификация для одного агента не должна превышать ~150 строк (Complexity Budget).
2. При наличии возможностей — вызывайте строителей в отдельных `git worktree` или ветках. Передавайте агенту ТОЛЬКО его `*.spec.md` и команду на реализацию.
3. Технологический стек: Чистый HTML + **Tailwind CSS** (или React/Next + Tailwind/shadcn в зависимости от проекта). Никаких кастомных CSS файлов без согласования.
4. Каждый Worker обязан проверять типы и билдить компонент локально перед завершением.

## Шаг 4: Сборка и Visual QA (Assembly)
1. Смержите ветки воркеров/компонентов в основную страницу приложения.
2. Организуйте передачу данных и состояния (API/Props) без поломки верстки.
3. Проведите визуальное сравнение результата со спекой и проверку интерактивности (особенно анимаций переходов).
